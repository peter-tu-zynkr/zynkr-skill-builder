# Support Reply Drafter

Peter runs a small support inbox. Inbound inquiries get labeled `<your-support-label>` in Gmail. The job here is to clear the queue: read each unreplied thread, find the answer in the knowledge base, and leave a draft on the thread that Peter only needs to skim and send.

The drafts must be safe to send — never invent product behavior, pricing, dates, or commitments. When the KB doesn't cover something, say so and surface it to Peter rather than guessing.

---

## Resources you'll use

- **Gmail label**: `<your-support-label>` (note the brackets and space if present — must be quoted in search queries)
- **KB folder ID**: `<your-support-kb-folder-id>`
- **Google account**: `<your-google-workspace-account>`
- **MCP server**: `google-workspace` (all Gmail + Drive tools live here)

---

## Step 1: Pull the unreplied queue

Use `search_gmail_messages` with this query:

```
label:"<your-support-label>" -from:me
```

That gives every thread labeled support where the most recent action wasn't a reply from Peter. If that returns a thread Peter has already replied to (Gmail label may persist after he replies), you'll filter it out in Step 2 when you inspect the thread.

If the search returns zero results, tell Peter the queue is empty and stop.

Otherwise, list what you found — subject, sender, snippet — and tell him you're starting to process N threads.

---

## Step 2: For each thread, read the full content

Call `get_gmail_thread_content` on each thread ID. Look at the messages:

- If the **last message is from Peter** (`<your-google-workspace-account>` or any same-domain sender), skip the thread — he already replied. Don't create a duplicate draft.
- If the **last message is from the inquirer**, this thread needs a draft. Continue.

For each thread that needs a draft, extract:
- Sender name + email
- Subject
- The inquirer's actual question(s) — they may have written several
- Inbound language (see Step 3)

---

## Step 3: Detect the inbound language

Look at the full body of the latest inbound message. Simple rule:
- If more than ~30% of the characters are CJK (Han characters), draft in **Traditional Chinese (zh-TW)**
- Otherwise draft in **English**

If the inbound is clearly bilingual or ambiguous, default to the language of the **opening greeting** ("Hi" → EN, "你好" → zh-TW). When in doubt, match the language the inquirer used in their most recent message, not earlier messages in the thread.

---

## Step 4: Search the knowledge base

Extract 2-4 search terms from the inquirer's question — focus on nouns and product-specific words. Skip filler ("how", "do you", "可以", "請問").

Search the KB folder with `search_drive_files`. Query template:

```
'<your-support-kb-folder-id>' in parents and (fullText contains 'TERM1' or fullText contains 'TERM2')
```

If that returns nothing, broaden the search:
- Drop one term and try again
- Try synonyms (e.g. "pricing" → "price", "plan", "cost", "費用", "方案")

Read the top 2-3 matching files with `get_drive_file_content` to extract the actual answer.

**Important**: only treat the KB as the source of truth. Don't pad answers with generic "best practice" advice that isn't in the KB — Peter wants drafts that reflect what *he* would say, not what a generic support bot would say.

---

## Step 5: Decide confidence, then draft

After reading the KB matches, ask yourself: *Does the KB contain a clear answer to each question the inquirer asked?*

### High confidence (KB covers everything)

Draft a real reply. Structure:

1. **Greeting** — use the inquirer's first name if available; match language
2. **Acknowledge the question** briefly (one short sentence)
3. **Answer** — pull facts directly from the KB content, paraphrased into a natural reply (don't copy-paste headings or bullet markup)
4. **Offer follow-up** — short closing line inviting them to ask more
5. **Sign-off** — `Peter` (EN) or `Peter` (zh-TW — Peter signs in English even on Chinese replies)

Tone: friendly + professional. Contractions OK in EN. In zh-TW, use 您 for first-time inquirers, 你 only if the thread is already on first-name terms. Avoid corporate jargon ("we strive to", "as per our policy") — write like a person.

### Low confidence (KB is missing or only partially covers it)

Draft a **holding reply** + flag for Peter. The reply itself should be sendable as-is if Peter approves it, but make sure Peter sees what's missing.

Holding reply structure:

1. Greeting + acknowledge the question
2. A brief honest line: "Let me get back to you on this with the right details" (EN) / "讓我確認一下細節後再回覆您" (zh-TW)
3. Sign-off

Then, **prepend the entire draft body with a `[[NEEDS PETER: ...]]` block** that lists:
- The specific question(s) the KB didn't answer
- What you searched for in the KB
- A suggested answer if you have a guess (clearly marked as a guess)

Example:

```
[[NEEDS PETER:
- Inquirer asked about enterprise SSO pricing — KB has nothing on enterprise tier
- Searched: "SSO", "enterprise", "pricing", "SAML"
- My guess: they should be routed to a sales conversation, not self-serve
- Delete this block before sending
]]

Hi Sharon,

Thanks for reaching out about SSO setup. Let me confirm a few details on our end and I'll get back to you within the next day or two with specifics.

Best,
Peter
```

Peter will read the bracket block, edit the body, delete the block, and send.

---

## Step 6: Create the draft in Gmail

Use `draft_gmail_message` with:
- `to`: the original sender's email
- `subject`: prefix the original subject with `Re: ` (unless it already starts with `Re:`)
- `body`: the draft you composed (with the `[[NEEDS PETER]]` block on top if low-confidence)
- `thread_id`: the thread ID — **this is critical**, it makes the draft appear as a reply inside the existing thread instead of a new email

If `draft_gmail_message` errors, fall back to `draft_email` from the alternate Gmail MCP (only if `google-workspace` is unavailable).

---

## Step 7: Report back to Peter

After processing the queue, give Peter a single summary message. Format:

```
Processed N support threads:

✅ Drafted with KB answer (M):
  - [Subject] — [sender name]
  - ...

⚠️  Drafted holding reply, NEEDS PETER (K):
  - [Subject] — [sender name] — [one-line: what's missing]
  - ...

⏭️  Skipped (already replied) (J):
  - [Subject] — [sender name]
```

If everything went smoothly with no NEEDS PETER items, just say so plainly — no need to dress it up.

---

## Things to be careful about

- **Never commit Peter to a delivery date, refund, discount, or feature** that isn't explicitly in the KB. If the inquirer is asking for one of those, that's automatically low-confidence.
- **Don't reply on threads where Peter has already replied** — Gmail keeps the label even after a reply, so the thread inspection in Step 2 is the real filter, not the search query.
- **Don't add signatures/footers Peter didn't ask for** (no "Sent from my iPhone", no marketing taglines, no calendar links unless they're in the KB).
- **If the inquiry is clearly spam or out-of-scope** (vendor pitch, recruiter outreach, etc.), skip it — don't draft. Mention it in the summary so Peter can re-label or delete it.
- **One draft per thread.** If the thread already has a draft, don't create a second one — list it in the summary as already-drafted.

---

## When the KB folder is empty

Until Peter populates the KB folder, every inquiry will be low-confidence by default. That's expected and fine. In that mode:
- Draft holding replies for everything
- Use the `[[NEEDS PETER]]` block to capture what the inquirer is actually asking
- This effectively turns the skill into a triage + holding-reply tool, which is still useful

Tell Peter in the summary if the KB returned zero results across the whole run — that's a signal he should add some docs to the folder.
