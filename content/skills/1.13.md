# AI Literacy Topic Selector

```bash
npx skills add https://github.com/peter-tu-zynkr/zynkr-skill-builder --skill ai-literacy-topic-selector
```

Pick what to write about for an AI-literacy article. This skill runs a 3-stage IPO pipeline: validate the writer's context, select a target professional role from the audience distribution, then generate three realistic workplace problems framed in AI-judgment terms (intention × knowledge × cognitive mode × initiative × flow). Use it as the front of the AI-literacy article workflow before handing off to `ai-literacy-article-writer` (1.11).

---

## Step 1 — Load the knowledge base

Read the reference from `{{SKILL_DIR}}/references/ai-judgment-rules-and-audience.md`. It contains both:
- **Tab 1: 判斷對錯的規則** — the canonical IPO classification (intention 3, knowledge 4, cognitive mode 3, initiative 2, flow 3) with works-well-with / does-not-work-with tables and the six known conflict patterns
- **Tab 2: 客群分佈** — distribution of professional roles in the AI-literacy audience (sorted by count)

If these files are missing, fall back to:
- Knowledge doc: `https://docs.google.com/document/d/1GHdmwKWIOpYGN7u7njMX9VjZOeoKkm4hcLkBAIu1B7A/edit`

Store the two tabs as `K1` (judgment rules) and `K2` (audience distribution) in working memory.

---

## Step 2 — Stage 1: Validate context (Input)

Ask the user for:
1. **Writing intent** — what kind of AI-literacy article? (tutorial / mental model / case study / cautionary tale / framework)
2. **Their knowledge** — what AI topic do they understand deeply enough to teach?
3. **Audience** — who's the intended reader? (general professionals / specific role / their existing newsletter list)
4. **Workflow signal** — do they already have a hook, headline, or rough idea?

Build a context packet:

```
Context Packet:
  Intent: <category>
  Knowledge Depth: <topic + level>
  Audience Hint: <if any>
  Existing Hook: <if any>
```

Validate against `K1` rules — does the writer's knowledge depth match the article intent? Examples:
- Writing a tutorial requires Instruction–Execution intent + Deductive cognitive mode → writer needs step-by-step expertise
- Writing a cautionary tale uses Context–Direction intent + System 1 — writer needs case examples, not theory

Output validation result: **`Ready`**, **`Ready with caveats`** (list the gaps), or **`Not yet`** (ask one targeted follow-up).

---

## Step 3 — Stage 2: Select target role (Process)

From `K2`, rank top roles by count and filter by which roles the writer can credibly speak to.

Default priority order (from `K2`):
1. 行政人員 (81)
2. 其它 (59) — skip, too generic
3. 創意工作者 (52)
4. 專案經理 (46)
5. 業務人員 (45)
6. 待業中 (41) — special case, frames as career-pivot articles
7. 工程師 (37)
8. 管理階層 (31)
9. 產品經理 (30)
10. 執行專員 (26)

Present **3 role candidates** ranked by:
- Audience size (from `K2`)
- Writer's credibility match (from validation in Stage 1)
- Strategic value (does this role appear underserved in prior content?)

Format:

```
Recommended target roles:
1. <role> — <count from K2> readers
   Why: <1 sentence connecting writer's knowledge to this role's daily work>
2. <role> — <count>
   Why: <...>
3. <role> — <count>
   Why: <...>

Which role should the article target? (1 / 2 / 3 / Other)
```

Wait for user selection. Store as `TARGET_ROLE`.

---

## Step 4 — Stage 3: Generate 3 workplace problems (Output)

For `TARGET_ROLE`, generate **exactly 3** realistic workplace problems where AI would actually help. Each problem must:

1. **Be specific and observable** — not "improve productivity" but "decoding cryptic Slack threads from cross-functional teams"
2. **Map to an AI-judgment combination from `K1`** — explicitly label intention × knowledge × cognitive mode × initiative × flow
3. **Pass the conflict check** — flag if the combination hits one of the six conflict patterns:
   - Memory Paradox (Contextual + Transactional)
   - Useless Setup (Context-Direction + Transactional)
   - Missing Teacher (AI-Driven + Analogical)
   - Over-Thinking (Context-Direction + Deductive)
   - Over-Engineering (Associative + Cyclical)
   - Hallucination Trap (General Knowledge + Real-Time)

Output format:

```
Problem 1: <one-sentence problem statement>
  Daily context: <2 sentences — when does this happen, what makes it painful>
  AI judgment profile:
    Intention: <Context-Direction / Q&A / Instruction-Execution>
    Knowledge: <General / Contextual / RAG / Tool-Based>
    Cognitive Mode: <System 1 / Pattern Matching / System 2>
    Initiative: <Human-Driven / AI-Driven>
    Flow: <Transactional / Conversational / Cyclical>
  Conflict check: <None / Flagged: <pattern name> — <how to avoid in the article>>
  Article angle: <1 sentence on how this becomes a compelling article hook>

(repeat for Problem 2 and Problem 3)
```

---

## Step 5 — Hand-off

Ask the user:

```
Which of these 3 problems would you like to develop into an article?
Once selected, I'll hand off the context packet + chosen problem + judgment profile to `ai-literacy-article-writer`.
```

When the user picks one, package the output for the next skill:

```
HANDOFF_PACKET:
  Context Packet (from Stage 1)
  Target Role: <role>
  Chosen Problem: <problem 1/2/3 content>
  AI Judgment Profile: <full profile>
  Article Angle: <hook>
```

---

## Rules

- Generate exactly 3 problems — not 2, not 5
- Every problem must have a full AI judgment profile (5 fields)
- Every problem must pass conflict check, or be explicitly flagged
- Never generate problems for "其它" — too generic, ask for a real role
- The writer's chosen problem becomes the input for 1.11 — do not skip the judgment profile, it grounds the article's framing
