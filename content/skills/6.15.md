# Skill Publish

```bash
npx skills add https://github.com/peter-tu-zynkr/zynkr-skill-builder --skill skill-publish
```

The **mid-pipeline intake** for skills that already exist as artifacts. Use this when a SKILL.md is in hand — freshly built via `/skill-creator`, downloaded from another repo, or hand-authored — and you want it categorised, dedup-checked against the existing pipeline, and pushed into the same GitHub Project that `/skill-sourcer` feeds.

> **Where this fits:** `/skill-sourcer` (curation, raw inputs) → **`/skill-publish` (this skill — built artifacts)** → `/skill-triager` (review queue → build dispatch).
>
> `/skill-sourcer` and `/skill-publish` are siblings. They land in the same pipeline and use the same subagents (classifier, deduplicator, proposer). The difference: `/skill-sourcer` extracts intent from raw text/URLs; `/skill-publish` reads it straight from a finished SKILL.md and skips the extraction step.

> **Source of truth:** the skills pipeline lives in GitHub Project `<your-skills-pipeline-project>` (e.g., `owner/N`). Every pipeline entry is a GitHub Issue in `peter-tu-zynkr/zynkr-skill-idea` added to the Project with custom fields (Pipeline Status, Keep, Category, Intake Source, Build *). The legacy Google Sheet is deprecated.

---

## Step 1 — Locate the built skill

Accept input in any of these forms:

- **Folder path** → expect a `SKILL.md` at the root (e.g., `skills/6-engineer/skill-publish/SKILL.md`)
- **Direct `SKILL.md` path** → read the file
- **Pasted SKILL.md content** → use as-is
- **GitHub URL** to a `SKILL.md` (raw or blob) → fetch it

If the user didn't provide an input, ask:

> "Where's the built skill? Drop a folder path, a SKILL.md path, a GitHub link, or paste the SKILL.md text directly."

Once located, confirm to the user what was detected and where it lives. If the artifact is **inside** `zynkr-skill-builder/skills/N-category/slug/`, note that the skill is already in-tree — this matters for Step 5's `Build Status` decision.

---

## Step 2 — Parse the SKILL.md (no extractor subagent)

Read the SKILL.md and parse:

- **Frontmatter:** `name`, `description`, `category` (if pre-set), `author`, `input`, `output`, `status`, `synergy`
- **Body intro:** the first paragraph below the H1 — usually the human-readable one-liner
- **Source / upstream URL:** look for an `npx skills add …` block, a `**Source**:` line, or a `**Upstream**:` line. If none, ask whether this is a Peter-authored skill (no upstream) or an external one (need the URL).

Build a structured extract equivalent to what `skill-extractor` would have produced:

```
name:        <frontmatter.name>
title:       <H1 or humanised name>
description: <frontmatter.description>
input:       <frontmatter.input>
output:      <frontmatter.output>
who it helps: <inferred from description — ask if unclear>
source URL:  <upstream URL or "Peter-authored, no upstream">
prior category: <frontmatter.category or null>
```

Display it and ask:

> **"This is what I read from the SKILL.md. Anything to correct before classifying?"**

Wait for confirmation. Edits at this stage are cheap; mis-classifications downstream are not.

---

## Step 3 — Classify into taxonomy (subagent)

Spawn the **skill-classifier** subagent at `../skill-sourcer/agents/classifier.md` with the confirmed extract.

If the SKILL.md had a `category:` value, pass it as the **prior** and ask the classifier to either confirm or override it (with reasoning). A human-set category is informative but not authoritative — the same taxonomy check that runs for sourced skills should run here.

The classifier returns:
- category number (0–9) and name
- confidence: high | medium | low
- runner-up (if confidence is medium/low) with reasoning

Display the result. If confidence is medium or low, **or** if the classifier disagrees with the prior `category:`, show the runner-up and ask the user to confirm the category before proceeding.

---

## Step 4 — Check pipeline for duplicates (subagent)

Spawn the **skill-deduplicator** subagent at `../skill-sourcer/agents/deduplicator.md` with the confirmed extract and classifier output.

It reads all items from GitHub Project `<your-skills-pipeline-project>` via `gh project item-list` and checks each one for overlap.

Verdicts: `exact_duplicate`, `near_duplicate`, `partial_overlap`, `new`.

**If `exact_duplicate` or `near_duplicate`:**
> "We already have **[skill name]** (Pipeline Status: [status], Keep: [keep]) which covers this. Here's how they overlap: [summary]."
> Ask: **"This is built locally — do you want to (a) replace the existing entry, (b) attach this artifact to the existing issue, or (c) skip?"** Default to (c) unless the user has a clear reason.

**If `partial_overlap`:**
> "Partial overlap with **[skill name]**: [summary]. Different enough to potentially stand alone."
> Ask: **"Proceed with a new proposal anyway?"**

**If `new`:**
> "No existing entry covers this. Proceeding to proposal."

---

## Step 5 — Add to pipeline (subagent)

Spawn the **skill-proposer** subagent at `../skill-sourcer/agents/proposer.md` with all outputs from previous steps, **plus the build-state context** below.

### Build-state context (this is what differs from `/skill-sourcer`)

The skill already exists as an artifact. Tell the proposer:

- **`Intake Source`** → `skill-publish` (not `skill-sourcer` — this distinguishes mid-pipeline intakes for later analytics)
- **`Build Repo`** → `zynkr-skill-builder` (always, since that's where built skills live)
- **`Build Target Path`** → resolved from the artifact's location:
  - If the folder is already in-tree at `skills/N-cat/slug/`, use that path.
  - If the artifact lives outside the repo, propose `N-{cat-slug}/{name-slug}` based on the classifier output and ask the user to confirm.
- **`Build Status`** → ask the user which fits:
  - `ready-to-ship` (default for skills built via `/skill-creator` or downloaded but not yet committed)
  - `shipped` (if the skill is already committed in-tree and live)
  - `testing` (if there's still work to do before it can ship)
- **`Built Skill URL`** → the GitHub URL once committed (path inside `peter-tu-zynkr/zynkr-skill-builder`), or empty if not yet committed. If empty, leave a note so the proposer can backfill on commit.
- **`Artifact`** → `skill-md-only` (vs the sourcer default of `issue-only`) — there's a real SKILL.md attached to this proposal.

### Proposer actions

1. Create a GitHub Issue in `peter-tu-zynkr/zynkr-skill-idea` titled `[Skill Proposal] {{category_number}} — {{title}}` with structured body, label `skill-proposal`. Include in the body:
   - A `**Source**:` line (upstream URL or `Peter-authored`)
   - A `**Built Skill URL**:` line (or `pending publish workflow` — filled in after Step 5b)
   - A `**Built via**: skill-publish` line
2. Add the issue to GitHub Project `<your-skills-pipeline-project>` with the custom fields above.
3. Display the issue URL and ask: **"Set Keep=yes to approve now, or leave as ? for later review."**

If the user approves immediately:

4. Update the Project item: `Keep=yes`, `Pipeline Status=approved`.
5. Add the `triage-ready` label to the issue (signal for `/skill-triager` to pick it up).
6. Optionally create a markdown idea record at `peter-tu-zynkr/zynkr-skill-idea/skills/approved/{{slug}}.md`. If created, set `Artifact=synced`.

## Step 5b — Fire publish-skill dispatch (lands the SKILL.md in the repo)

After approval, the SKILL.md still needs to physically land in `peter-tu-zynkr/zynkr-skill-builder` at `skills/<N-cat>/<slug>/SKILL.md`. Don't do this by hand — fire the `publish-skill` workflow.

**If the artifact already lives at a GitHub URL** (e.g. another repo, a gist, or a raw URL), use `skill_md_url`:

```bash
gh api repos/peter-tu-zynkr/zynkr-skill-builder/dispatches \
  -X POST \
  -f event_type=skill-publish-request \
  -F "client_payload[issue_number]=<num>" \
  -F "client_payload[issue_repo]=peter-tu-zynkr/zynkr-skill-idea" \
  -F "client_payload[slug]=<slug>" \
  -F "client_payload[category]=<category-number-or-slug>" \
  -F "client_payload[skill_md_url]=<raw or blob URL to the SKILL.md>"
```

**If the artifact only lives locally** (e.g. just produced by `/skill-creator`), inline the content as base64:

```bash
gh api repos/peter-tu-zynkr/zynkr-skill-builder/dispatches \
  -X POST \
  -f event_type=skill-publish-request \
  -F "client_payload[issue_number]=<num>" \
  -F "client_payload[issue_repo]=peter-tu-zynkr/zynkr-skill-idea" \
  -F "client_payload[slug]=<slug>" \
  -F "client_payload[category]=<category-number-or-slug>" \
  -F "client_payload[skill_md_b64]=$(base64 -w0 < /path/to/SKILL.md)"
```

Exactly one of `skill_md_url` or `skill_md_b64` must be set — the workflow rejects payloads that supply both or neither.

**Verify the workflow landed:**

```bash
gh run list --repo peter-tu-zynkr/zynkr-skill-builder \
  --workflow publish-skill.yml --limit 3
```

Surface the run URL and (once the workflow finishes) the resulting PR URL to the user. The workflow:

1. Writes `skills/<N-cat>/<slug>/SKILL.md` (idempotent — refuses to overwrite an existing file)
2. Validates frontmatter against the schema
3. Creates branch `skill/<slug>` and opens a PR titled `publish(<slug>): from peter-tu-zynkr/zynkr-skill-idea#<num>`
4. Comments back on the source issue with the PR URL

After the PR merges, `ingest-skills.yml` runs and publishes the skill to the marketplace within ~30 s.

**Backfill the issue body:** once the PR is open, edit the issue's `**Built Skill URL**:` line to point at the merged URL `https://github.com/peter-tu-zynkr/zynkr-skill-builder/blob/main/<path>`. Update the Project's `Built Skill URL` field to match.

## Step 6 — Sync READMEs

Run this step after every approval.

### A — Local README (skill tree)

The skill tree in `skills/README.md` is auto-generated by `scripts/build-taxonomy-tree.ts`. If the artifact is now committed in-tree:

```bash
npm run build:taxonomy-tree    # or: npx tsx scripts/build-taxonomy-tree.ts
```

Verify the new entry appears under the right category before continuing.

If the artifact isn't committed yet, skip this step and leave a reminder for the user to run it after commit.

### B — `SOURCED.md` in `zynkr-skill-idea`

Append a row, same schema as `/skill-sourcer` uses. Add a column note that this came in via `skill-publish` (in the `Source` cell or as a trailing tag), so the two intake paths stay distinguishable in the index.

Use `gh api` to read the current `SOURCED.md`, append the row, and PUT back. If the file doesn't exist, create it with the header.

### C — Build handoff tracking

Same model as `/skill-sourcer`:

- `Pipeline Status`: `proposed` → `approved` → (handed to `/skill-triager`) → `building` → `shipped`
- `Build Status`: starts at whatever you set in Step 5 (`ready-to-ship` / `shipped` / `testing`), evolves from there.

### Triager handoff (Option D — `confirm-ship`)

`/skill-publish` items are designed to flow through `/skill-triager`'s **`confirm-ship`** branch, not `assign-build`. The triager picks up the `triage-ready` label, recognises `Intake Source=skill-publish`, and runs three read-only verifications:

1. `gh api contents` to confirm the SKILL.md is in `main`
2. `generated/skills-index.json` contains the slug (i.e., `ingest-skills.yml` already ran)
3. `https://www.zynkr.ai/api/skills` returns the slug (i.e., the Supabase mirror has synced)

If all three pass, the triager flips `Pipeline Status` + `Build Status` → `shipped`, swaps the label to `shipped`, and closes the issue. **No `repository_dispatch` is fired** — the artifact is already live by the time triage runs, so triage is a verification + bookkeeping step here, not orchestration.

Tell the user this on completion of Step 5 so they know what to expect when they next run `/skill-triager`.

## Step 7 — Done

Summarise:

- Where the artifact lived
- What category it landed in (and whether the classifier overrode the prior)
- Duplicate verdict
- Issue created (URL)
- Project membership set with Build Status / Built Skill URL
- Markdown idea record (if approved and created)
- Whether `build:taxonomy-tree` was re-run

**Completion checklist (all must be ticked before done):**
- [ ] Artifact located and SKILL.md parsed
- [ ] Extract confirmed by user
- [ ] Category confirmed (with sign-off if confidence was medium/low or overrode the prior)
- [ ] Dedup check completed against the pipeline Project
- [ ] If proceeding: issue created with `skill-proposal` label and `**Built via**: skill-publish` in the body
- [ ] Project item set: `Intake Source=skill-publish`, `Build Repo=zynkr-skill-builder`, `Build Target Path` set, `Build Status` set, `Artifact=skill-md-only` (or `synced`)
- [ ] If approved: `Keep=yes`, `Pipeline Status=approved`, `triage-ready` label added
- [ ] `publish-skill` dispatch fired (Step 5b)
- [ ] `publish-skill.yml` workflow run observed and PR URL surfaced
- [ ] `Built Skill URL` backfilled in the issue body and the Project field after PR opens
- [ ] If approved: `SOURCED.md` row appended (after PR merges, optional)

Ask: **"Want to publish another skill?"**

## Error handling

- **SKILL.md missing frontmatter fields:** ask the user to fill the gap (especially `name`, `description`) before classifying — the classifier needs the description.
- **Artifact path doesn't exist:** stop and re-prompt for a valid location.
- **Classifier disagrees with the SKILL.md `category:`:** never silently override — surface the disagreement and let the user pick.
- **Dedup returns near/exact and user picks "attach":** post a comment on the existing issue with the artifact path / URL and the `**Built via**: skill-publish` line; do not create a second issue.
- **Proposer dispatch fails (non-2xx from GitHub):** roll back any label/field changes already applied so the artifact can be retried cleanly.
