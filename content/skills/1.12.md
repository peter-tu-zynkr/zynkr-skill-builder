# AI Literacy Article Writer

```bash
npx skills add https://github.com/peter-tu-zynkr/zynkr-skill-builder --skill ai-literacy-article-writer
```

Write a publish-ready AI-literacy article from a context packet. This skill assumes upstream work (1.10 `ai-literacy-topic-selector`) has already done the topic selection and judgment profiling. Its job is to turn that packet into a first-person article that reads like real working experience and ends with reusable prompts the reader can copy. Use it as the second half of the AI-literacy article pipeline.

---

## Step 1 — Receive the context packet

Expected input from `ai-literacy-topic-selector` (or hand-assembled):

```
Context Packet:
  Intent: <article category>
  Audience: <target role + size hint>
  Pain Point: <specific workplace problem>
  Constraints: <length, tone, channel>
  Desired Outcome: <what the reader should be able to do after reading>
  AI Judgment Profile:
    Intention: ...
    Knowledge: ...
    Cognitive Mode: ...
    Initiative: ...
    Flow: ...
  Article Angle: <hook>
```

If the packet is incomplete, ask **one** targeted question to fill the most critical gap. Do not run the full intake again — that's 1.10's job.

Store the packet as `CONTEXT`.

---

## Step 2 — Diagnose reader cognition

Before writing, infer:
1. **Reader's current mental model** — what do they assume about AI today?
2. **The cognitive gap** — what's the gap between their assumption and the reality your article reveals?
3. **The pattern to break** — which assumption needs to crack first?

Most AI-literacy articles fail because they explain instead of disrupt. Identify the disruption point.

Output (internal — don't show user yet):

```
Reader Model:
  Current assumption: <belief>
  Gap: <what they don't realize>
  Pattern to break: <single sentence>
```

---

## Step 3 — Reframe the problem

Take the pain point from `CONTEXT` and reframe it in AI-judgment language. Use the profile to find the angle:

- If profile is **Instruction-Execution + Deductive + Cyclical** → reframe as "you're treating this as one-shot when it's a refinement loop"
- If profile is **Q&A + Contextual + System 1** → reframe as "you're asking AI to remember but the session forgot"
- If profile is **Context-Direction + Pattern Matching + Conversational** → reframe as "you're describing instead of showing"

The reframe becomes the article's spine.

Output (internal):

```
Reframe: <one sentence that flips the reader's mental model>
```

---

## Step 4 — Craft the article using pattern-break → reframe → copy-paste

### Section A — Pattern-break (the hook, 100–150 words)

Open with a specific, observable moment from the pain point. First-person. Concrete. No abstractions.

**Don't:** "Many professionals struggle with AI productivity..."
**Do:** "上週三晚上 11 點，我盯著一份 50 頁的提案，AI 給的摘要全是廢話。"

The hook ends with the broken pattern named explicitly: "我以為 AI 該怎樣，結果完全不對。"

### Section B — Reframe (the insight, 300–400 words)

Walk the reader through:
1. Why the assumed pattern doesn't work (use the conflict pattern from `CONTEXT` if applicable — e.g., Memory Paradox, Hallucination Trap)
2. The actual AI-judgment combination that fits this problem
3. A short story or analogy that lands the reframe

Stay first-person. Use the writer's voice. Avoid bullet points here — this section reads as narrative.

### Section C — Copy-paste (the reusable prompts, 300–500 words)

Provide **2–3 ready-to-use prompts** the reader can paste into ChatGPT/Claude immediately. Each prompt:
- Has a clear name (e.g., `用蘇格拉底式提問拆解一份提案`)
- Specifies the AI judgment combination it uses (1 line)
- Includes the actual prompt text in a code block
- Shows expected output format (1–2 lines)

Format:

````
### Prompt 1: <name>

**用法：** <judgment combo in plain words, e.g., "instruction + deductive thinking + cyclical refinement">

```
You are a <role>. I'll give you <input>.
Walk me through:
1. ...
2. ...
3. ...
Output as <format>.
```

**輸出範例：** <2-line expected output>
````

### Section D — Closing (50–100 words)

Tie back to the opening moment. Don't summarize the article. Instead, invite the reader to try one of the prompts and report back. End with a concrete, low-friction action.

---

## Step 5 — Quality checks

Before delivery, verify:
- [ ] Total length matches `CONTEXT.Constraints.length` (typically 800–1200 words for zh-TW)
- [ ] First-person used consistently — no "many people" or "users"
- [ ] At least one specific, observable moment in the hook (time, place, sensory detail)
- [ ] Every reusable prompt has a name + judgment combo + working code block + expected output
- [ ] Conflict pattern from `CONTEXT` is explicitly named if relevant
- [ ] CTA is one concrete action, not a generic "let me know what you think"

---

## Step 6 — Output

Render the article in zh-TW (default) or English (if `CONTEXT.Audience` is English-speaking). Format:

```markdown
# <title — written last, derived from the reframe>

<Section A — pattern-break>

<Section B — reframe>

<Section C — 2–3 copy-paste prompts>

<Section D — closing>

---

*想試試看？把上面任一個 prompt 貼到 ChatGPT 或 Claude，跑完跟我說發生什麼。*
```

After delivery, ask:

```
Want to:
1. Generate an SEO title pack for this article (→ hand to 1.05 article-title-suggester)
2. Run reader-critique on the draft (→ hand to 1.04 reader-perspective)
3. Edit a specific section
```

---

## Rules

- First-person only — no "many professionals", no "users"
- Pattern-break section must include a specific moment (time, place, or sensory detail)
- Every copy-paste prompt must be runnable as-is in ChatGPT/Claude — no `[fill in here]` placeholders unless the placeholder is the point
- Reframe section uses narrative, not bullets — bullets are for Section C only
- Never invent statistics or studies; if a claim needs evidence, mark `[need source]` for the writer to fill in
- Title is written last, after the article body is complete — derive it from the reframe sentence
