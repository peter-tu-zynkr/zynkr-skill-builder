# Development Process

> This document describes Zynkr's end-to-end development process for AI assistant projects, from intake to governance. The process is operationalized via the **Consultant service Kanban** on [Notion](https://www.notion.so/281ad634eef28067ac0dc616bf494b75).

## Process Overview

Each project follows a standardized 7-stage pipeline, tracked as a Kanban board with swimlane statuses:

```
Sales outbound → Consult intake → Discovery/Shadowing → Design/Visualize → Proposal → Develop → Deploy → Govern → Closed
```

| Stage | Group | Owner | Key Output |
|---|---|---|---|
| Sales outbound | To Do | Peter Tu + Jane Liao | Qualified lead |
| Consult intake | To Do | Peter Tu + Jane Liao | Intake form, initial scoping |
| Discovery / Shadowing | In Progress | Project lead | Pain Point Heatmap, as-is process map (SIPOC) |
| Design / Visualize | In Progress | Project lead | FE/BE/DB swimlane diagram (Lucidchart) |
| Proposal | In Progress | Project lead | Solution architecture, scope & deliverables |
| Develop | In Progress | Peter Tu (AI dev) | AI assistant bundle (prompts + knowledge + tests) |
| Deploy | Complete | Project lead | Production rollout, onboarding |
| Govern | Complete | Project lead | Marketplace listing, ongoing monitoring |
| Closed / Archive | Complete | — | — |

## Stage Details

### 1. Sales Outbound / Consult Intake

Identify and qualify opportunities.

* Prospecting, inbound form, lead qualification
* Initial scoping conversation

### 2. Discovery / Shadowing

Understand the problem and map what exists.

* **Prospecting:** Identify which process to analyze — look for high-traffic, high-friction, or high-impact workflows
* **Discovery:** Interview process owners, review SOPs/logs/tools, document friction points and workarounds
* **Shadowing:** Observe actual operations, compare "as-documented" vs. "actual" flows, validate with operators
* **Process Mapping (SIPOC):** Create Suppliers-Inputs-Process-Outputs-Customers diagram to define scope, stakeholders, and dependencies
* **Deliverables:** Pain Point Heatmap, ground-truth as-is process, SIPOC diagram

### 3. Design / Visualize

Redesign the end-to-end experience. Full methodology in `4.0 training-design/8-week-bootcamp/course-outline-en/Ch5`.

#### Workflow Visualization Convention

Swimlane diagrams use a standardized shape + color system in Lucidchart:

**Shapes** (semantic, color-agnostic):

| Shape | Meaning |
|---|---|
| Terminal (Oval) | Start / End / Outcome boundary |
| Process (Rectangle) | A step that performs work |
| Decision (Diamond) | Branching gate (yes/no or multi-branch) |
| Document (Wavy) | Document-like / knowledge artifact |
| Database (Cylinder) | Durable system-of-record storage |
| Manual I/O (Slanted) | Human inputs or outputs via UI |
| Manual Operation (Trapezoid) | Human execution step (HITL) |
| Off-page Link (Pentagon) | Cross-workflow/page connector |
| Merge/Junction (Crosshair) | Join/fork control |

**Colors** (ownership/execution/storage):

| Color | Meaning |
|---|---|
| Red | FE surface (human interaction) |
| Light Blue | Backend manual process (HITL execution) |
| Purple | Backend deterministic automation (state machine/rules) |
| Dark Blue | Backend modern automation (LLM/agentic) |
| Green | DB durable (system of record) |
| Light Green | Ephemeral context (session-only) |
| Dark Green | RAG-related knowledge |

#### Architecture Framework — FE / BE / DB

| Layer | Meaning | Design Questions |
|---|---|---|
| Front End (FE) | Human-facing — user actions, UI, triggers | Who initiates? What's visible? What's the input? |
| Back End (BE) | Logic/automation — what happens behind the scenes | What rules transform the input? What's the output? |
| Database (DB) | Data storage — where knowledge/data is stored | What knowledge is referenced, updated, or retrieved? |

#### Process Optimization

* I-P-O validation (each step has clear Input-Process-Output)
* Dependency mapping (handoffs, shared data, triggers)
* Friction elimination (duplicative steps, unnecessary loops, missing steps)
* Sequencing for control & throughput (cheap checks upstream, expensive work downstream)

#### Automation Feasibility Assessment

Two scorecards (1-3 per dimension):

**Digital Readiness Scorecard:**
* Digital touchpoints (1=manual/offline, 2=digital but unstructured, 3=structured capture)
* Standardization (1=judgment-heavy, 2=pattern-based, 3=rule-based)
* Data availability (1=fragmented, 2=partially available, 3=traceable+usable)
* Knowledge readiness (1=missing, 2=exists but not operationalized, 3=operational+indexed)
* Score 10-12: automation-ready | 7-9: partial + prep | 4-6: digitize/codify first

**Workflow Selection Scorecard:**
* AI fit per step (Objective x Traffic)
* Integration readiness (1=no integration, 2=lightweight, 3=full API gateway)
* Risk & compliance (1=high risk, 2=medium, 3=low risk)
* ROI (1=low, 2=medium, 3=high)
* Score 10-12: automate now | 7-9: partial + prerequisites | 4-6: redesign first

#### Modularization

* Identify repeating logic (approval, notification, data validation, execution)
* Abstract into modular objects with I-P-O contracts
* Label with metadata: function, owner, data source, trigger type

* **Deliverables:** FE/BE/DB swimlane diagram, Digital Readiness Scorecard, Workflow Selection Scorecard, modular process library

### 4. Proposal

Package the solution.

* Solution architecture document
* Scope, timeline, deliverables
* Validated with SIPOC Agent

### 5. Development

Build AI assistants node by node. Full methodology in `4.0 training-design/8-week-bootcamp/course-outline-en/Ch6`.

#### Workflow-first Placement

* Mark each step as FE / BE / DB ownership
* Insert assistant nodes where they reduce cost or ambiguity (drafting, summarization, classification, routing, evidence packaging, comparison)
* Define decision authority per node: assistant proposes → deterministic validates → human approves (if needed)

#### Node Contract Sheet (per assistant)

* **Input:** required fields, evidence requirements, source-of-truth, missing-input behavior
* **Process:** mode (LLM/deterministic/HITL), tools/RAG allowed, verification rules, stop conditions
* **Output:** schema/format, acceptance criteria, escalation path

#### Routine Typing — 5D Tag

| Dimension | Options |
|---|---|
| Why | Context-Direction / Q-A / Instruction-Execution |
| What | General / Contextual / RAG / Tool-based |
| How | Associative / Analogical / Deductive |
| Who | Human-driven / AI-driven |
| When | Transactional / Conversational / Cyclical |

Check for known conflicts (e.g., Contextual + Transactional = memory paradox). Fix by changing one dimension or adding one containment mechanism.

#### Prompt Engineering — 3-Layer Fold

| Layer | Purpose | Stability |
|---|---|---|
| A) Control Layer | Identity, output contract, constraints, quality bar, safety | Stable (reusable) |
| B) Node Logic | 5D routine, I/O rules, tool permissions, guardrails, verification, stop conditions | Node-specific |
| C) Run Payload | User intent, inputs, retrieved artifacts, constraints, output fields | Per-run |

#### Handoff Payload Spec

What passes to the next step:
* Verified facts (with evidence pointers)
* Decisions made + rationale category
* Open questions + required next inputs
* Status flags (confidence / missing info / escalation)

#### Testing

Per node: 3 test cases (normal / edge / missing input). Fix via contract change, one-dimension move (5D), or containment upgrade.

* **Deliverable per node:** Node bundle = workflow slot + contract sheet + routine tag + folded prompt package (A/B/C) + handoff payload spec + 3 tests

### 6. Deploy

Ship to production and ensure adoption.

* UAT and staged rollout
* Onboarding and training
* Change management

### 7. Govern

Keep it running and listed.

* Knowledge management: finding the right answer, surfacing at the right time, keeping answers current
* Marketplace listing and maintenance on the [AI index](https://docs.google.com/spreadsheets/d/1QvrtRB0_1tHM67IEq01T-UpKtsv4nXtIkWB3k5xynd4/edit?gid=0#gid=0)
* Feedback collection, usage monitoring, iterative improvement
